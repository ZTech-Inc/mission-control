**In src/lib/org-scanner.ts -- Fix upsertAssignment (line 340-348):** Replace the `ON CONFLICT` clause in `upsertAssignment` to guard against overwriting manual promotions. Change from:

```sql
ON CONFLICT(workspace_id, agent_id, team_external_id) DO UPDATE SET
  role = excluded.role,
  assigned_at = excluded.assigned_at,
  source = excluded.source
```

To:

```sql
ON CONFLICT(workspace_id, agent_id, team_external_id) DO UPDATE SET
  role = CASE WHEN agent_team_assignments.source = 'manual' THEN agent_team_assignments.role ELSE excluded.role END,
  assigned_at = excluded.assigned_at,
  source = CASE WHEN agent_team_assignments.source = 'manual' THEN 'manual' ELSE excluded.source END
```

This ensures filesystem rescans preserve `role` and `source` for manually-promoted leads.

**In src/lib/org-scanner.ts -- Propagate manager_agent_id in snapshot:** After `applyFilesystemOrgPersistence` is called (around line 515) and before the return statement (line 517), add code to read back `manager_agent_id` from the DB and annotate the departments array:

```typescript
// Annotate departments with persisted manager_agent_id
const leadRows = db.prepare(
  `SELECT external_id, manager_agent_id FROM departments WHERE workspace_id = ?`
).all(workspaceId) as Array<{ external_id: number; manager_agent_id: number | null }>
const leadMap = new Map(leadRows.map((row) => [row.external_id, row.manager_agent_id]))
for (const dept of departments) {
  const leadId = leadMap.get(dept.id)
  if (leadId != null) dept.manager_agent_id = leadId
}
```

This requires `getDatabase()` to be accessible in `scanFilesystemOrg` -- it already is since `applyFilesystemOrgPersistence` uses it internally. Import `getDatabase` at the top of `scanFilesystemOrg` or use the same `db` instance. Since `applyFilesystemOrgPersistence` creates its own `db` internally, the cleanest approach is to call `getDatabase()` directly after the persistence call.
